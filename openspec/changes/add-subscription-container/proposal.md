## Why

The chart deploys only the panel backend today, leaving the subscription-page sibling service (`remnawave/subscription-page`) as an explicit follow-up from the original chart proposal. Without it, Kubernetes operators either run the subscription page outside the cluster or hand-roll its manifests, losing the "bundled" experience that docker-compose users get out of the box. Users subscribing via links generated by the panel also lose the branded, localized rendered page that the subscription-page service provides — the panel's own `/api/sub` endpoint serves raw subscription data, not the UI.

## What Changes

- Add an opt-in subscription-page sidecar container (`remnawave/subscription-page`) to the panel `Deployment`'s pod template, gated on `subscription.enabled`. Co-located sidecar keeps the "bundled" topology from docker-compose and lets the subscription page talk to the panel over `http://localhost:<APP_PORT>` without a new `Service`.
- Surface the subscription-page's environment-variable configuration (`APP_PORT`, `REMNAWAVE_PANEL_URL`, `REMNAWAVE_API_TOKEN`, `CUSTOM_SUB_PREFIX`, `MARZBAN_LEGACY_LINK_ENABLED`, `MARZBAN_LEGACY_SECRET_KEY`, `CADDY_AUTH_API_TOKEN`) through `values.yaml` under a `subscription.*` group, with defaults that match the upstream bundled example.
- Add a new subscription port to the existing panel `Service` (named `subscription`), and route external traffic via a new subscription-specific `Ingress` host or `HTTPRoute` that coexists with the panel's front-end under the same mutual-exclusion rule.
- Store the panel API token in the chart-managed `Secret` (or require `REMNAWAVE_API_TOKEN` in `existingSecret` when `subscription.enabled: true`), with the same two-track pattern already used for JWT secrets.
- **MODIFIED**: When `subscription.enabled: true`, derive `SUB_PUBLIC_DOMAIN` from the subscription-page's ingress/gateway host (plus optional prefix) instead of the panel's. When disabled, derivation behaves exactly as today.
- Extend the existing `checksum/config` and `checksum/secret` pod-template annotations to cover the subscription-page's config/secret keys, so changes roll the pod.
- Add a Helm install guide section covering the bundled subscription-page scenario in [docs/install/](docs/install/).

## Capabilities

### New Capabilities

None. The subscription-page is an extension of the existing chart, not a new capability boundary. All new requirements belong to the `helm-chart` capability.

### Modified Capabilities

- `helm-chart`: Adds requirements for the subscription-page sidecar (conditional rendering, container port, env-var surface, secret contract), extends the `Service` to carry the subscription port, adds a new subscription routing front-end that shares the ingress/gateway mutual-exclusion rule, and modifies `SUB_PUBLIC_DOMAIN` derivation so it resolves to the subscription-page's host when the sidecar is enabled.

## Impact

- **Chart templates**: [charts/remnawave-panel/templates/deployment.yaml](charts/remnawave-panel/templates/deployment.yaml) (sidecar container block), [service.yaml](charts/remnawave-panel/templates/service.yaml) (new port), [configmap.yaml](charts/remnawave-panel/templates/configmap.yaml) and [secret.yaml](charts/remnawave-panel/templates/secret.yaml) (new keys), [ingress.yaml](charts/remnawave-panel/templates/ingress.yaml) and [httproute.yaml](charts/remnawave-panel/templates/httproute.yaml) (subscription host/route), [_helpers.tpl](charts/remnawave-panel/templates/_helpers.tpl) (subscription domain derivation), and [values.schema.json](charts/remnawave-panel/values.schema.json) (new `subscription.*` keys).
- **values.yaml**: New `subscription.*` block with sidecar toggle, image ref, env vars, routing sub-block, and resources. The existing `subscription.publicDomain` and `subscription.publicPath` keys stay backward-compatible — they still win when explicitly set.
- **Documentation**: New section in [docs/install/helm-chart.md](docs/install/helm-chart.md) covering the bundled subscription-page topology, corresponding to [docs/install/subscription-page/bundled.md](docs/install/subscription-page/bundled.md) for compose.
- **No upstream changes**: `remnawave/subscription-page` is consumed as-is. No app code changes.
- **No breaking change to existing chart users**: `subscription.enabled` defaults to `false`. Existing installs continue to behave identically after the upgrade.
- **Out of scope**: a separate `Deployment` for the subscription page (the sidecar pattern is deliberately chosen — see design.md), HPA, and the "separate-server" topology from [docs/install/subscription-page/separate-server.md](docs/install/subscription-page/separate-server.md) (operators wanting that can run a second `helm install` or a different chart).
